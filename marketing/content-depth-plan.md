# 콘텐츠 깊이 플랜 — 48개 툴 "최상 품질" 만들기

목표: 48개 툴 페이지 전부에 깊은 편집 콘텐츠 추가. 양적 디렉토리
(Futurepedia 2000+ × 얕은 페이지)와 SEO에서 다른 게임을 하기 위함.

**현실 인식:** 47개 × ~700단어 = 약 33,000단어. 한 번에 다 못 함.
10개씩 5배치로 "차근차근". 각 배치 = 1~2주 페이스.

---

## 0. 인프라 현황 (이미 완성된 것)

조사 결과 단계 1(인프라)은 거의 다 돼 있음:

| 인프라 | 상태 |
|---|---|
| 콘텐츠 그릇 (`lib/tool-extended-content.ts`) | ✅ overviewHtml · useCases · pricingDetail · faq |
| `/tool/[slug]` 렌더링 | ✅ 4개 섹션 전부 표시 |
| SoftwareApplication JSON-LD | ✅ 자동 |
| BreadcrumbList JSON-LD | ✅ 자동 |
| FAQPage JSON-LD | ✅ faq 있으면 자동 (리치 결과 자격) |
| 가짜 평점 제외 (정책 준수) | ✅ |
| "Full review coming soon" 폴백 | ✅ 콘텐츠 없을 때 |

→ **추가 인프라 작업 거의 불필요.** 콘텐츠만 채우면 됨.

### 선택적 인프라 폴리시 (단계 0, ~1시간, 콘텐츠 시작 전 한 번)

작게 추가하면 E-E-A-T(전문성 신호) + 신선도 ↑:

1. **`lastReviewed` 날짜 필드** — 인터페이스에 추가, 페이지에 "Last reviewed:
   June 2026" 뱃지 + JSON-LD `dateModified`. 신선도 = Google 랭킹 신호.
2. **`makerInfo` 필드** (선택) — 회사/설립연도. "누가 만들었나" E-E-A-T.
3. **"Alternatives" 자동 섹션** (선택) — 같은 카테고리 3개를
   `/compare/X-vs-Y` 링크로. 내부 링크 = 크롤 흐름 + 비교 페이지 트래픽.

이 셋은 콘텐츠 품질과 독립적이라 나중에 해도 됨. 우선순위 낮음.

## 1. 한 툴당 "최상 품질" 정의

Lovable(현재 유일한 완성 예시)을 기준선으로:

| 필드 | 분량 | 핵심 |
|---|---|---|
| **overviewHtml** | 500~700단어, 4~5문단 | 진짜 리뷰. 무엇이고 / 어떻게 작동하고 / 강점 / **솔직한 약점** / 누구한테 맞고 안 맞나. 내부 링크 2~3개 (`/tool/X`, `/compare/X-vs-Y`) |
| **useCases** | 3개, 각 60~80단어 | 구체적 사용 시나리오 (페르소나 + 실제 작업) |
| **pricingDetail** | 1~2문단 | Free/Paid 등급 너머의 실제 가격 + **가격 함정** (크레딧 소진, 시트당 과금 등) |
| **faq** | 4~6개 Q&A | 실제 검색되는 질문. "X vs Y?", "free로 충분?", "누구한테 맞나" |

### 품질을 가르는 3가지 (AI 티 제거)

1. **솔직한 cons** — 공식 페이지가 절대 안 쓰는 것. "10개 화면 넘으면
   모델이 충돌하는 변경을 시작한다" 같은 구체적 한계.
2. **"누구한테 안 맞나"** — 부정 추천. 신뢰의 핵심.
3. **본인 실사용 한 줄** ⭐ — "직접 써보니 [구체적 관찰]". 이게 있으면
   AI-generated 디렉토리와 결정적으로 갈림. 모든 툴에 못 넣어도, 본인이
   써본 것엔 반드시.

## 2. 작업 분담 (효율)

| 단계 | 담당 | 내용 |
|---|---|---|
| 초안 작성 | **나** | overviewHtml + useCases + pricingDetail + faq 풀 초안. 솔직한 cons / 누구한테 안 맞나 포함 |
| 사실 검증 | **나** | 가격/기능은 WebSearch로 2026년 6월 기준 확인 |
| 검토 + 실사용 | **너** | 읽고 "여기 틀림" / "더 짧게" 가벼운 편집 + 써본 툴엔 실사용 한 줄 ⭐ |
| 머지 | **나** | 파일에 반영 → 배치 단위 커밋 → 배포 |

페이스: **한 배치(10개)당 1~2주.** 본인은 배치당 검토 1~2시간만.

## 4. 배치 1 상세 실행 (다음 세션 시작점)

다음에 작업 시작하면 이 순서:

1. **나**: 배치 1의 10개 각각 WebSearch로 현재 가격/기능 확인 (2026년 6월)
2. **나**: 10개 풀 콘텐츠 초안 작성 → `lib/tool-extended-content.ts`에 추가
   하는 형태로, 또는 검토용 마크다운으로 먼저 제시
3. **너**: 검토. 특히 본인이 써본 것 (chatgpt/claude/cursor/notion 등 대부분
   써봤을 듯) 에 실사용 한 줄 ⭐
4. **나**: 반영 → 커밋 → 배포 → 10개 페이지 "Full review coming soon" 사라짐
5. **너**: GSC에 배치 1의 10개 인덱싱 요청 (하루 한도 딱 맞음)

**검토 부담 줄이는 옵션:** 한 번에 10개가 많으면 배치를 5개씩 쪼개도 됨.
"차근차근" 정신에 맞춰 본인 페이스로.

## 5. 측정 (배치별)

각 배치 배포 + 인덱싱 2~3주 후:

- **GSC Pages**: 해당 10개가 "Indexed"로 넘어갔는지
- **GSC Performance**: 해당 툴명 + "X vs Y" 쿼리 노출 시작됐는지
- **체류 시간** (Vercel/GA): 깊은 콘텐츠 페이지의 평균 체류가 얕은 페이지보다
  긴지 — 품질 전략이 작동하는지의 증거

배치 1 결과가 좋으면 (노출 증가) 나머지 배치 가속. 안 좋으면 콘텐츠 각도
조정 후 진행 (추측 아닌 데이터 기반).

## 7. 이 플랜의 솔직한 한계

- **10주는 길다.** 본인 검토 시간이 병목. 더 빨리 가려면 본인이 실사용
  한 줄을 포기하고 내 초안을 가볍게만 검토 → 품질 약간 희생, 속도 ↑.
- **콘텐츠만으로 트래픽이 폭발하진 않음.** 깊은 콘텐츠는 "인덱싱되고
  순위 오를 자격"을 만들 뿐. 여전히 도메인 권위(Phase 1 백링크)와 병행해야
  실제 순위가 오름. 콘텐츠 + 백링크는 곱셈 관계.
- **48개도 여전히 작다.** 대형 디렉토리 대비 절대 검색량은 작음. 우리의
  베팅은 "검색량 작아도 전환/체류 강한 niche 권위 사이트". 이건 트래픽
  total보다 질을 택한 의도된 선택 — 본인이 이미 결정한 방향.
